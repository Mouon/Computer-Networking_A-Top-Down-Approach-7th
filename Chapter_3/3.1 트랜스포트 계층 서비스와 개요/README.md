# 3.1 트랜스포트 계층 서비스와 개요

3장에서는 **트랜스포트 계층(transport layer)** 을 다룬다.

앞 장의 애플리케이션 계층에서는 HTTP, SMTP, DNS 같은 프로토콜을 봤다.

그런데 애플리케이션 프로세스가 실제로 네트워크를 통해 메시지를 주고받으려면, 애플리케이션 계층 아래에서 데이터를 옮겨주는 계층이 필요하다.

그 역할을 하는 것이 바로 트랜스포트 계층이다.

## 3.1.1 트랜스포트 계층의 역할

트랜스포트 계층은 서로 다른 종단 시스템에서 동작하는 **애플리케이션 프로세스 사이의 논리적 통신(logical communication)** 을 제공한다.

여기서 논리적 통신이라는 말이 중요하다.

실제로 데이터는 라우터, 링크, 스위치 등을 지나며 물리적으로 이동한다.

하지만 애플리케이션 입장에서는 마치 두 프로세스가 직접 연결된 것처럼 보인다.

예를 들어 내 브라우저 프로세스와 어떤 웹 서버 프로세스가 HTTP 메시지를 주고받는다고 하자.

브라우저는 중간에 몇 개의 라우터가 있는지, 링크 계층에서 어떤 일이 일어나는지 직접 신경 쓰지 않는다.

브라우저는 소켓에 데이터를 쓰고, 서버 프로세스는 자기 소켓에서 데이터를 읽는다.

이때 두 프로세스 사이의 통신을 만들어주는 핵심 계층이 트랜스포트 계층이다.

## 3.1.2 트랜스포트 계층과 네트워크 계층의 차이

트랜스포트 계층과 네트워크 계층은 모두 데이터를 이동시키지만, 제공하는 통신 범위가 다르다.

### 네트워크 계층

네트워크 계층은 **호스트와 호스트 사이의 통신** 을 담당한다.

즉, 한 종단 시스템에서 다른 종단 시스템까지 데이터그램을 전달하는 역할을 한다.

인터넷에서는 IP가 이 역할을 한다.

IP는 목적지 호스트까지 패킷을 보내려고 노력하지만, 애플리케이션 프로세스까지 직접 구분해주지는 않는다.

### 트랜스포트 계층

트랜스포트 계층은 **프로세스와 프로세스 사이의 통신** 을 담당한다.

하나의 호스트 안에는 여러 애플리케이션 프로세스가 동시에 실행될 수 있다.

예를 들어 한 컴퓨터에서 브라우저, 메신저, 메일 앱이 동시에 네트워크를 사용할 수 있다.

네트워크 계층이 데이터를 목적지 호스트까지 가져오면, 트랜스포트 계층은 이 데이터를 어떤 프로세스에게 전달해야 하는지 결정한다.

이때 사용되는 것이 **포트 번호(port number)** 이다.

정리하면 다음과 같다.

| 계층 | 통신 단위 | 대표 역할 |
| --- | --- | --- |
| 네트워크 계층 | 호스트 간 통신 | 목적지 호스트까지 패킷 전달 |
| 트랜스포트 계층 | 프로세스 간 통신 | 목적지 호스트 안의 올바른 프로세스로 전달 |

## 3.1.3 트랜스포트 계층은 어디에 구현될까?

트랜스포트 계층은 **종단 시스템(end system)** 에 구현된다.

라우터는 일반적으로 트랜스포트 계층 프로토콜을 처리하지 않는다.

라우터는 네트워크 계층의 IP 데이터그램을 보고 다음 홉으로 전달한다.

반면 송신 호스트와 수신 호스트는 트랜스포트 계층 세그먼트를 만들고 해석한다.

송신 측에서는 다음과 같은 일이 일어난다.

1. 애플리케이션 프로세스가 소켓으로 메시지를 보낸다.
2. 트랜스포트 계층이 메시지를 세그먼트(segment)로 만든다.
3. 세그먼트에 헤더 정보를 붙인다.
4. 네트워크 계층으로 내려보낸다.

수신 측에서는 반대로 동작한다.

1. 네트워크 계층이 IP 데이터그램을 트랜스포트 계층으로 올린다.
2. 트랜스포트 계층이 세그먼트 헤더를 확인한다.
3. 어떤 소켓으로 전달해야 하는지 결정한다.
4. 해당 애플리케이션 프로세스에게 데이터를 전달한다.

## 3.1.4 인터넷의 트랜스포트 프로토콜

인터넷은 대표적으로 두 가지 트랜스포트 계층 프로토콜을 제공한다.

1. **UDP(User Datagram Protocol)**
2. **TCP(Transmission Control Protocol)**

두 프로토콜은 모두 IP 위에서 동작한다.

하지만 애플리케이션에게 제공하는 서비스는 크게 다르다.

### UDP

UDP는 매우 단순한 트랜스포트 프로토콜이다.

UDP는 다음과 같은 특징을 가진다.

- 비연결형이다.
- 신뢰적인 데이터 전송을 보장하지 않는다.
- 혼잡 제어를 제공하지 않는다.
- 세그먼트 손실, 중복, 순서 뒤바뀜이 발생할 수 있다.
- 헤더가 작고 동작이 단순하다.

UDP는 애플리케이션에게 IP의 서비스 모델을 거의 그대로 보여준다고 볼 수 있다.

즉, 최소한의 기능만 제공한다.

하지만 바로 이 단순함 때문에 DNS, 실시간 스트리밍, 일부 게임, 음성 애플리케이션에서 UDP가 사용될 수 있다.

### TCP

TCP는 UDP보다 훨씬 많은 기능을 제공한다.

TCP는 다음과 같은 특징을 가진다.

- 연결 지향형이다.
- 신뢰적인 데이터 전송을 제공한다.
- 순서대로 데이터를 전달한다.
- 흐름 제어를 제공한다.
- 혼잡 제어를 제공한다.

TCP는 데이터가 손실되면 재전송하고, 순서가 뒤바뀌면 올바른 순서로 애플리케이션에게 전달한다.

그래서 웹, 파일 전송, 이메일처럼 데이터의 정확성이 중요한 애플리케이션에서 많이 사용된다.

## 3.1.5 트랜스포트 계층이 제공할 수 있는 서비스

트랜스포트 계층은 애플리케이션에게 여러 종류의 서비스를 제공할 수 있다.

인터넷의 TCP와 UDP가 모든 서비스를 제공하는 것은 아니지만, 개념적으로는 다음 서비스를 생각할 수 있다.

### 신뢰적인 데이터 전송

네트워크에서는 패킷이 손실되거나 손상될 수 있다.

트랜스포트 계층은 이런 문제를 감지하고 복구해서 애플리케이션에게 오류 없는 데이터를 전달할 수 있다.

TCP가 이 기능을 제공한다.

UDP는 이 기능을 제공하지 않는다.

### 처리율 보장

어떤 애플리케이션은 최소 처리율이 필요하다.

예를 들어 고화질 비디오 스트리밍은 일정 수준 이상의 대역폭이 필요하다.

하지만 인터넷의 TCP와 UDP는 애플리케이션에게 특정 처리율을 보장하지 않는다.

### 시간 보장

실시간 음성이나 게임에서는 지연 시간이 매우 중요하다.

예를 들어 100ms 안에 데이터가 도착해야 자연스럽게 통신할 수 있는 경우가 있다.

하지만 인터넷의 TCP와 UDP는 지연 시간을 보장하지 않는다.

### 보안

트랜스포트 계층 근처에서 암호화와 인증을 제공할 수도 있다.

대표적으로 TLS는 TCP 위에서 동작하며 애플리케이션 데이터에 기밀성과 무결성을 제공한다.

## 3.1.6 세그먼트

트랜스포트 계층의 패킷을 **세그먼트(segment)** 라고 부른다.

애플리케이션 메시지가 트랜스포트 계층으로 내려오면, 트랜스포트 계층은 여기에 헤더를 붙여 세그먼트를 만든다.

세그먼트에는 보통 다음과 같은 정보가 포함된다.

- 출발지 포트 번호
- 목적지 포트 번호
- 오류 검출 정보
- TCP의 경우 순서 번호, 확인응답 번호, 윈도우 크기 등

이 세그먼트는 다시 IP 데이터그램 안에 캡슐화되어 네트워크 계층으로 전달된다.

## 정리

트랜스포트 계층은 애플리케이션 프로세스 사이의 논리적 통신을 제공한다.

네트워크 계층이 호스트 간 통신을 담당한다면, 트랜스포트 계층은 호스트 안의 프로세스까지 구분해서 데이터를 전달한다.

인터넷의 대표 트랜스포트 프로토콜은 UDP와 TCP이다.

UDP는 단순하고 비신뢰적이며, TCP는 연결 지향적이고 신뢰적인 데이터 전송을 제공한다.

이후 절에서는 먼저 다중화와 역다중화를 보고, UDP와 신뢰적 데이터 전송 원리를 거쳐 TCP를 자세히 살펴본다.
