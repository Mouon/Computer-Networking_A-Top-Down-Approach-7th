# 3.5 연결지향형 트랜스포트: TCP

이번 절에서는 인터넷에서 가장 중요한 트랜스포트 프로토콜 중 하나인 **TCP(Transmission Control Protocol)** 를 다룬다.

TCP는 애플리케이션에게 신뢰적인 데이터 전송 서비스를 제공한다.

웹, 이메일, 파일 전송처럼 데이터가 정확하게 도착해야 하는 서비스에서 TCP가 널리 사용된다.

## 3.5.1 TCP 연결

TCP는 **연결 지향형(connection-oriented)** 프로토콜이다.

UDP처럼 바로 데이터를 보내는 것이 아니라, 데이터를 보내기 전에 클라이언트와 서버가 먼저 연결을 설정한다.

이 연결 설정 과정에서 양쪽은 앞으로 데이터를 주고받기 위한 상태 정보를 준비한다.

TCP 연결은 실제 회선이 물리적으로 생긴다는 뜻은 아니다.

두 종단 시스템의 TCP가 서로 상태를 유지하고, 그 상태를 기반으로 데이터를 주고받는다는 의미이다.

라우터들은 TCP 연결 상태를 유지하지 않는다.

### 전이중 서비스

TCP 연결은 **전이중(full-duplex)** 서비스를 제공한다.

전이중이라는 것은 양쪽이 동시에 데이터를 보낼 수 있다는 뜻이다.

예를 들어 클라이언트가 서버에게 HTTP 요청을 보내는 동안, 서버도 클라이언트에게 데이터를 보낼 수 있다.

### 점대점 연결

TCP 연결은 항상 두 종단 사이의 점대점 연결이다.

하나의 송신자와 하나의 수신자 사이에 만들어진다.

멀티캐스트처럼 한 송신자가 여러 수신자에게 동시에 보내는 구조는 TCP 연결 모델과 맞지 않는다.

### MSS

TCP는 애플리케이션으로부터 받은 데이터를 적절한 크기의 조각으로 나누어 세그먼트에 담는다.

TCP 세그먼트에 담을 수 있는 애플리케이션 데이터의 최대 크기를 **MSS(Maximum Segment Size)**라고 한다.

MSS는 보통 링크 계층의 MTU를 고려해 정해진다.

## 3.5.2 TCP 세그먼트 구조

TCP 세그먼트는 헤더와 데이터로 구성된다.

TCP 헤더에는 신뢰적인 전송과 연결 관리를 위한 다양한 필드가 들어 있다.

대표적인 필드는 다음과 같다.

### 출발지 포트 번호와 목적지 포트 번호

애플리케이션 프로세스를 식별하기 위해 사용한다.

TCP의 다중화와 역다중화에서 핵심적인 필드이다.

### 순서 번호

TCP는 데이터를 바이트 스트림으로 본다.

순서 번호(sequence number)는 세그먼트에 담긴 첫 번째 바이트의 번호를 의미한다.

예를 들어 어떤 세그먼트의 순서 번호가 100이고 데이터가 500바이트라면, 이 세그먼트는 100번부터 599번 바이트를 담고 있다고 볼 수 있다.

### 확인응답 번호

확인응답 번호(acknowledgment number)는 수신자가 다음에 받기를 기대하는 바이트 번호이다.

예를 들어 수신자가 0번부터 999번 바이트까지 정상적으로 받았다면, ACK 번호는 1000이 된다.

이는 "1000번 바이트부터 보내줘"라는 뜻이다.

TCP의 ACK는 기본적으로 **누적 확인응답(cumulative ACK)** 방식이다.

### 헤더 길이

TCP 헤더는 옵션 필드를 포함할 수 있으므로 길이가 고정되어 있지 않다.

헤더 길이 필드는 TCP 헤더가 어디까지인지 알려준다.

### 플래그 필드

TCP 헤더에는 여러 제어 플래그가 있다.

대표적인 것은 다음과 같다.

- `SYN`: 연결 설정에 사용
- `FIN`: 연결 종료에 사용
- `ACK`: 확인응답 번호가 유효함을 나타냄
- `RST`: 연결 재설정
- `PSH`: 데이터를 즉시 애플리케이션으로 밀어 올리도록 요청
- `URG`: 긴급 데이터 표시

### 수신 윈도우

수신 윈도우(receive window)는 흐름 제어에 사용된다.

수신자가 현재 얼마나 더 받을 수 있는지를 송신자에게 알려준다.

### 체크섬

TCP 체크섬은 세그먼트에 비트 오류가 있는지 검출하는 데 사용된다.

## 3.5.3 왕복 시간 추정과 타임아웃

TCP는 손실된 세그먼트를 재전송해야 한다.

그런데 언제 재전송해야 할까?

너무 빨리 재전송하면 실제로는 손실되지 않은 세그먼트를 불필요하게 다시 보낼 수 있다.

너무 늦게 재전송하면 손실 복구가 느려진다.

그래서 TCP는 **RTT(Round-Trip Time)**를 추정하고, 이를 기반으로 타임아웃 시간을 정한다.

### SampleRTT

`SampleRTT`는 어떤 세그먼트를 보낸 시점부터 그 세그먼트에 대한 ACK를 받을 때까지 걸린 시간이다.

하지만 SampleRTT는 네트워크 상황에 따라 계속 변한다.

그래서 TCP는 SampleRTT 하나만 보고 타임아웃을 정하지 않는다.

### EstimatedRTT

TCP는 여러 SampleRTT를 부드럽게 평균내기 위해 `EstimatedRTT`를 사용한다.

대표적인 계산식은 다음과 같다.

```text
EstimatedRTT = (1 - α) * EstimatedRTT + α * SampleRTT
```

보통 `α = 0.125`가 사용된다.

이 방식은 최근 측정값을 반영하면서도 급격한 변동을 완화한다.

### DevRTT

RTT가 얼마나 흔들리는지도 중요하다.

TCP는 RTT 변동성을 나타내기 위해 `DevRTT`를 계산한다.

```text
DevRTT = (1 - β) * DevRTT + β * |SampleRTT - EstimatedRTT|
```

보통 `β = 0.25`가 사용된다.

### TimeoutInterval

최종 타임아웃 시간은 대략 다음과 같이 정한다.

```text
TimeoutInterval = EstimatedRTT + 4 * DevRTT
```

RTT 변동이 크면 타임아웃 시간을 더 길게 잡고, 변동이 작으면 더 짧게 잡을 수 있다.

## 3.5.4 신뢰적인 데이터 전송

TCP는 신뢰적인 데이터 전송을 제공한다.

이를 위해 TCP는 다음 기능들을 사용한다.

- 순서 번호
- 확인응답
- 타이머
- 재전송
- 누적 ACK

### TCP의 데이터 전송 방식

TCP는 애플리케이션 데이터를 바이트 스트림으로 다룬다.

송신 TCP는 이 바이트 스트림을 세그먼트로 나누어 보낸다.

수신 TCP는 받은 바이트들을 순서대로 맞춰 애플리케이션에게 전달한다.

### 누적 ACK

TCP는 누적 ACK를 사용한다.

ACK 번호는 다음에 기대하는 바이트 번호이다.

예를 들어 수신자가 0~999번 바이트를 받았다면 ACK 1000을 보낸다.

만약 1000~1499번 바이트를 담은 세그먼트가 손실되고, 1500~1999번 바이트가 먼저 도착했다면 수신자는 여전히 ACK 1000을 보낼 수 있다.

이는 1000번부터의 데이터를 아직 기다리고 있다는 뜻이다.

### 타임아웃 기반 재전송

송신자는 세그먼트를 보낸 뒤 ACK를 기다린다.

타임아웃이 발생하면 해당 세그먼트를 재전송한다.

이 방식은 손실을 복구할 수 있지만, 타임아웃까지 기다려야 하므로 느릴 수 있다.

### 중복 ACK와 빠른 재전송

TCP는 손실을 더 빠르게 감지하기 위해 **중복 ACK(duplicate ACK)**를 활용한다.

수신자가 순서가 맞지 않는 세그먼트를 받으면, 마지막으로 순서대로 받은 바이트 다음 번호를 ACK로 반복해서 보낼 수 있다.

송신자가 같은 ACK를 여러 번 받으면 어떤 세그먼트가 손실되었을 가능성이 높다고 판단한다.

일반적으로 같은 ACK를 3번 중복해서 받으면 타임아웃을 기다리지 않고 손실된 것으로 보이는 세그먼트를 재전송한다.

이를 **빠른 재전송(fast retransmit)**이라고 한다.

## 3.5.5 흐름 제어

TCP는 신뢰적인 전송뿐만 아니라 **흐름 제어(flow control)**도 제공한다.

흐름 제어의 목적은 송신자가 수신자의 처리 능력보다 너무 빠르게 데이터를 보내지 않도록 하는 것이다.

수신 호스트에는 TCP 수신 버퍼가 있다.

애플리케이션이 버퍼에서 데이터를 읽어가는 속도보다 TCP가 데이터를 받는 속도가 빠르면 버퍼가 가득 찰 수 있다.

버퍼가 가득 찬 상태에서 더 많은 데이터가 도착하면 손실이 발생할 수 있다.

TCP는 이를 막기 위해 수신자가 송신자에게 현재 남은 버퍼 공간을 알려준다.

이 값이 **수신 윈도우(receive window, rwnd)**이다.

### 수신 윈도우

수신 윈도우는 수신자가 현재 더 받을 수 있는 데이터 양을 나타낸다.

송신자는 미확인 데이터 양이 수신 윈도우를 넘지 않도록 조절한다.

간단히 말하면 다음 조건을 지키는 것이다.

```text
LastByteSent - LastByteAcked <= rwnd
```

이 조건을 지키면 송신자는 수신자의 버퍼를 넘치게 하지 않는다.

### 흐름 제어와 혼잡 제어의 차이

흐름 제어와 혼잡 제어는 비슷해 보이지만 목적이 다르다.

- 흐름 제어: 수신자의 버퍼를 보호한다.
- 혼잡 제어: 네트워크 전체의 혼잡을 줄인다.

3.5절에서는 흐름 제어를 다루고, 혼잡 제어는 이후 3.6~3.7절에서 더 자세히 다룬다.

## 3.5.6 TCP 연결 관리

TCP는 데이터를 보내기 전에 연결을 설정하고, 데이터 전송이 끝나면 연결을 종료한다.

### 3-way handshake

TCP 연결 설정은 보통 **3-way handshake**로 이루어진다.

과정은 다음과 같다.

1. 클라이언트가 서버에게 `SYN` 세그먼트를 보낸다.
2. 서버가 `SYNACK` 세그먼트로 응답한다.
3. 클라이언트가 `ACK` 세그먼트를 보낸다.

이 과정이 끝나면 양쪽은 TCP 연결이 설정되었다고 판단하고 데이터를 주고받을 수 있다.

3-way handshake는 양쪽이 서로 통신 가능하다는 것을 확인하고, 초기 순서 번호를 교환하는 역할을 한다.

### 연결 종료

TCP 연결 종료에는 `FIN` 세그먼트가 사용된다.

한쪽이 더 이상 보낼 데이터가 없으면 `FIN`을 보낸다.

상대방은 이에 대해 ACK를 보내고, 자신도 종료할 준비가 되면 FIN을 보낸다.

최종적으로 양쪽이 FIN과 ACK를 주고받으면 연결이 닫힌다.

### SYN Flood 공격

TCP 연결 설정 과정은 공격에 악용될 수 있다.

공격자가 많은 SYN 세그먼트를 보내고 마지막 ACK를 보내지 않으면, 서버는 반쯤 열린 연결 상태를 많이 유지해야 한다.

이런 공격을 **SYN Flood 공격**이라고 한다.

이를 완화하기 위해 SYN cookie 같은 기법이 사용될 수 있다.

## 3.5.7 TCP를 보는 큰 그림

TCP는 여러 기능이 결합된 프로토콜이다.

단순히 "신뢰적인 전송" 하나만 제공하는 것이 아니다.

TCP는 다음을 함께 제공한다.

- 연결 설정과 종료
- 바이트 스트림 기반 전송
- 순서 번호와 ACK
- 손실 감지와 재전송
- 흐름 제어
- 혼잡 제어

이 중 3.5절에서는 연결, 세그먼트 구조, RTT 추정, 신뢰적 전송, 흐름 제어, 연결 관리를 다뤘다.

혼잡 제어는 3.6절 이후에서 이어진다.

## 정리

TCP는 연결 지향형 트랜스포트 프로토콜이다.

데이터를 바이트 스트림으로 보고, 순서 번호와 ACK를 사용해 신뢰적인 전송을 제공한다.

RTT를 추정해 타임아웃을 정하고, 타임아웃 또는 중복 ACK를 통해 손실된 세그먼트를 재전송한다.

또한 수신 윈도우를 사용해 흐름 제어를 수행하고, 3-way handshake로 연결을 설정한다.

TCP는 복잡하지만, 그만큼 많은 인터넷 애플리케이션이 의존하는 핵심 프로토콜이다.
