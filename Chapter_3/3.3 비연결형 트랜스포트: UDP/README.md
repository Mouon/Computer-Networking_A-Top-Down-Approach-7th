# 3.3 비연결형 트랜스포트: UDP

이번 절에서는 인터넷의 대표적인 비연결형 트랜스포트 프로토콜인 **UDP(User Datagram Protocol)** 를 살펴본다.

UDP는 TCP와 비교하면 매우 단순하다.

하지만 단순하다는 것이 쓸모없다는 뜻은 아니다.

UDP는 연결 설정 없이 빠르게 데이터를 보내고 싶을 때 유용하다.

## 3.3.1 UDP의 기본 특징

UDP는 **비연결형(connectionless)** 프로토콜이다.

TCP처럼 데이터를 보내기 전에 연결을 설정하지 않는다.

클라이언트는 UDP 소켓을 만들고, 목적지 IP 주소와 포트 번호를 지정해서 바로 세그먼트를 보낸다.

UDP는 다음과 같은 특징을 가진다.

- 연결 설정 과정이 없다.
- 신뢰적인 데이터 전송을 제공하지 않는다.
- 순서 보장을 제공하지 않는다.
- 흐름 제어를 제공하지 않는다.
- 혼잡 제어를 제공하지 않는다.
- 헤더 크기가 작다.

UDP는 IP 위에 아주 얇은 기능만 얹은 프로토콜이라고 볼 수 있다.

IP가 호스트 간 전달을 제공한다면, UDP는 여기에 포트 번호를 이용한 프로세스 간 전달과 오류 검출 정도를 추가한다.

## 3.3.2 UDP를 사용하는 이유

TCP가 신뢰적인 데이터 전송을 제공한다면, 왜 굳이 UDP를 사용할까?

이유는 몇 가지가 있다.

### 연결 설정 지연이 없다

TCP는 데이터를 보내기 전에 연결을 설정해야 한다.

반면 UDP는 연결 설정 없이 바로 데이터를 보낼 수 있다.

짧은 요청과 응답을 주고받는 애플리케이션에서는 이 차이가 중요할 수 있다.

대표적으로 DNS는 UDP를 많이 사용한다.

### 구현이 단순하다

UDP는 연결 상태를 유지하지 않는다.

서버는 각 클라이언트에 대해 TCP 연결 상태를 관리하지 않아도 된다.

이 때문에 서버 구현이 단순해지고, 많은 클라이언트의 짧은 요청을 처리하기 좋다.

### 헤더 오버헤드가 작다

UDP 헤더는 8바이트이다.

TCP 헤더는 기본적으로 20바이트이다.

작은 메시지를 자주 보내는 경우 헤더가 작다는 점이 장점이 될 수 있다.

### 혼잡 제어에 의해 전송 속도가 제한되지 않는다

TCP는 네트워크 혼잡을 감지하면 전송 속도를 낮춘다.

반면 UDP는 자체적으로 혼잡 제어를 하지 않는다.

실시간 음성, 영상, 게임처럼 일정한 전송률을 유지하고 싶은 애플리케이션은 UDP를 선택할 수 있다.

물론 이것은 조심해야 한다.

UDP 애플리케이션이 너무 많은 데이터를 보내면 네트워크 혼잡을 악화시킬 수 있다.

## 3.3.3 UDP가 사용되는 예

UDP는 다음과 같은 애플리케이션에서 자주 사용된다.

- DNS
- 실시간 음성 통화
- 실시간 영상 스트리밍
- 온라인 게임
- 네트워크 관리 프로토콜

다만 오늘날에는 신뢰성, 보안, 혼잡 제어를 애플리케이션 계층에서 직접 구현한 뒤 UDP를 사용하는 경우도 많다.

예를 들어 QUIC은 UDP 위에서 동작하지만, TCP와 TLS가 제공하던 여러 기능을 자체적으로 구현한다.

## 3.3.4 UDP 세그먼트 구조

UDP 세그먼트는 매우 간단한 구조를 가진다.

UDP 헤더는 다음 4개의 필드로 구성된다.

```text
출발지 포트 번호 | 목적지 포트 번호
길이             | 체크섬
애플리케이션 데이터
```

각 필드는 2바이트씩이다.

따라서 UDP 헤더의 크기는 총 8바이트이다.

### 출발지 포트 번호

응답을 받을 때 필요한 포트 번호이다.

예를 들어 클라이언트가 DNS 서버에게 요청을 보낼 때, 서버는 이 출발지 포트 번호를 보고 응답을 돌려준다.

### 목적지 포트 번호

수신 호스트에서 어떤 프로세스에게 데이터를 전달할지 결정하는 데 사용된다.

예를 들어 DNS 서버는 보통 53번 포트를 사용한다.

### 길이

UDP 세그먼트의 전체 길이를 나타낸다.

헤더와 데이터를 모두 포함한 길이이다.

### 체크섬

세그먼트에 오류가 있는지 검사하기 위해 사용된다.

## 3.3.5 UDP 체크섬

UDP 체크섬은 전송 중 비트 오류가 발생했는지 확인하기 위한 기능이다.

송신 측은 UDP 세그먼트의 여러 16비트 단어를 더하고, 그 결과를 이용해 체크섬 값을 만든다.

수신 측은 받은 세그먼트에 대해 같은 계산을 수행한다.

계산 결과가 맞지 않으면 세그먼트에 오류가 있다고 판단한다.

체크섬은 오류를 검출하는 기능이지, 오류를 복구하는 기능은 아니다.

즉, UDP는 오류가 있는 세그먼트를 발견할 수는 있지만, TCP처럼 자동으로 재전송해서 복구하지 않는다.

복구가 필요하다면 애플리케이션이 직접 처리해야 한다.

## 3.3.6 UDP의 한계

UDP는 단순하지만 그만큼 많은 것을 보장하지 않는다.

UDP를 사용할 때 애플리케이션 개발자는 다음 문제를 고려해야 한다.

### 손실 가능성

UDP 세그먼트는 중간 라우터의 버퍼가 가득 차거나 오류가 발생하면 손실될 수 있다.

UDP는 손실된 세그먼트를 자동으로 재전송하지 않는다.

### 순서 뒤바뀜

여러 UDP 세그먼트가 서로 다른 경로를 지나면 도착 순서가 바뀔 수 있다.

UDP는 순서를 맞춰서 애플리케이션에게 전달하지 않는다.

### 중복 가능성

네트워크 상황에 따라 중복된 데이터가 도착할 가능성도 고려해야 한다.

UDP 자체는 중복 제거 기능을 제공하지 않는다.

### 혼잡 제어 부재

UDP는 네트워크가 혼잡해도 자동으로 전송 속도를 줄이지 않는다.

그래서 많은 UDP 애플리케이션이 동시에 과도한 트래픽을 보내면 네트워크 전체에 나쁜 영향을 줄 수 있다.

## 3.3.7 UDP를 사용할 때의 관점

UDP는 "대충 보내도 되는 프로토콜"이 아니다.

정확히는 **기본 기능만 제공하고 나머지는 애플리케이션이 책임지는 프로토콜** 에 가깝다.

애플리케이션이 자체적으로 다음 기능을 구현할 수도 있다.

- 손실 복구
- 순서 재정렬
- 중복 제거
- 혼잡 제어
- 암호화

이런 방식은 구현이 어려워질 수 있지만, 애플리케이션 목적에 맞는 전송 방식을 직접 설계할 수 있다는 장점이 있다.

## 정리

UDP는 비연결형 트랜스포트 프로토콜이다.

연결 설정이 없고 헤더가 작으며 동작이 단순하다.

하지만 신뢰성, 순서 보장, 흐름 제어, 혼잡 제어를 제공하지 않는다.

UDP는 DNS처럼 짧은 요청/응답에 적합하고, 실시간 애플리케이션처럼 지연이 중요한 경우에도 사용될 수 있다.

UDP를 사용할 때는 애플리케이션이 필요한 신뢰성이나 제어 기능을 직접 설계해야 한다.
