# 3.2 다중화와 역다중화

트랜스포트 계층의 가장 기본적인 역할 중 하나는 **다중화(multiplexing)**와 **역다중화(demultiplexing)**이다.

이 기능 덕분에 하나의 호스트에서 여러 네트워크 애플리케이션이 동시에 동작할 수 있다.

예를 들어 한 컴퓨터에서 브라우저로 웹 페이지를 보고, 메신저를 사용하고, 백그라운드에서 파일을 다운로드한다고 해보자.

네트워크로 들어오는 세그먼트들은 모두 같은 호스트에 도착한다.

그렇다면 운영체제는 이 데이터를 어떤 애플리케이션 프로세스에게 전달해야 할까?

이 문제를 해결하는 것이 역다중화이다.

## 3.2.1 소켓과 포트 번호

트랜스포트 계층은 애플리케이션 프로세스에게 직접 데이터를 주는 것이 아니라 **소켓(socket)**을 통해 데이터를 전달한다.

소켓은 애플리케이션 프로세스와 네트워크 사이의 출입구이다.

한 호스트 안에는 여러 소켓이 있을 수 있다.

트랜스포트 계층은 세그먼트 헤더를 보고 어떤 소켓으로 데이터를 넘겨야 하는지 결정한다.

이때 핵심적으로 사용되는 값이 **포트 번호(port number)**이다.

포트 번호는 16비트 값이므로 범위는 다음과 같다.

```text
0 ~ 65535
```

이 중 일부 포트는 잘 알려진 애플리케이션에 예약되어 있다.

예를 들면 다음과 같다.

| 포트 번호 | 프로토콜/서비스 |
| --- | --- |
| 20, 21 | FTP |
| 25 | SMTP |
| 53 | DNS |
| 80 | HTTP |
| 443 | HTTPS |

서버 애플리케이션은 보통 잘 알려진 포트에서 클라이언트의 요청을 기다린다.

클라이언트는 서버의 IP 주소와 포트 번호를 알아야 해당 서버 프로세스와 통신할 수 있다.

## 3.2.2 역다중화

**역다중화(demultiplexing)**는 수신 호스트의 트랜스포트 계층이 들어온 세그먼트를 올바른 소켓으로 전달하는 과정이다.

수신 호스트에는 여러 소켓이 있을 수 있다.

트랜스포트 계층은 세그먼트 헤더의 정보를 확인하고, 해당 세그먼트를 어느 소켓에 전달할지 결정한다.

예를 들어 수신 호스트에 다음 프로세스들이 실행 중이라고 하자.

- 웹 서버
- SSH 서버
- DNS 서버

각 서버는 서로 다른 포트에서 요청을 기다린다.

세그먼트의 목적지 포트가 80이면 웹 서버 소켓으로, 53이면 DNS 서버 소켓으로 전달된다.

## 3.2.3 다중화

**다중화(multiplexing)**는 송신 호스트에서 여러 소켓으로부터 데이터를 모아 세그먼트로 만들고, 네트워크 계층으로 내려보내는 과정이다.

송신 측에서는 여러 애플리케이션 프로세스가 동시에 데이터를 보낼 수 있다.

트랜스포트 계층은 각 데이터에 헤더를 붙인다.

헤더에는 출발지 포트 번호와 목적지 포트 번호 같은 정보가 들어간다.

이 정보가 있어야 수신 측에서 역다중화를 할 수 있다.

## 3.2.4 UDP의 역다중화

UDP 소켓은 보통 다음 두 값으로 식별된다.

```text
(목적지 IP 주소, 목적지 포트 번호)
```

UDP 세그먼트가 호스트에 도착하면, 수신 호스트는 세그먼트의 목적지 포트 번호를 확인한다.

그리고 해당 포트에 연결된 UDP 소켓으로 데이터를 전달한다.

중요한 점은 UDP에서는 출발지 IP 주소나 출발지 포트 번호가 달라도, 목적지 포트가 같으면 같은 소켓으로 전달될 수 있다는 것이다.

예를 들어 어떤 DNS 서버가 53번 포트에서 UDP 요청을 기다리고 있다고 하자.

여러 클라이언트가 서로 다른 IP와 포트에서 요청을 보내도, 모두 서버의 53번 포트 소켓으로 들어온다.

서버는 응답을 보낼 때 요청 세그먼트에 들어 있던 출발지 IP 주소와 출발지 포트 번호를 사용한다.

## 3.2.5 TCP의 역다중화

TCP 소켓은 UDP보다 더 많은 정보로 식별된다.

TCP 연결 소켓은 다음 네 가지 값으로 식별된다.

```text
(출발지 IP 주소, 출발지 포트 번호, 목적지 IP 주소, 목적지 포트 번호)
```

이 네 값을 **4-튜플(4-tuple)**이라고 생각할 수 있다.

TCP 서버는 하나의 포트에서 여러 클라이언트의 연결을 동시에 처리할 수 있다.

예를 들어 웹 서버는 80번 포트에서 요청을 기다린다.

클라이언트 A와 클라이언트 B가 같은 서버의 80번 포트에 접속해도, 두 연결은 서로 다른 TCP 연결로 구분된다.

왜냐하면 두 연결의 출발지 IP 주소나 출발지 포트 번호가 다르기 때문이다.

즉, TCP에서는 목적지 포트만으로 소켓을 구분하지 않고, 네 가지 값을 함께 사용한다.

## 3.2.6 TCP 서버의 환영 소켓과 연결 소켓

TCP 서버에는 두 종류의 소켓을 생각할 수 있다.

1. **환영 소켓(welcome socket)**
2. **연결 소켓(connection socket)**

환영 소켓은 클라이언트의 연결 요청을 기다리는 소켓이다.

웹 서버가 80번 포트에서 연결 요청을 기다릴 때, 이 소켓이 환영 소켓 역할을 한다.

클라이언트가 연결을 요청하면 서버는 새로운 연결 소켓을 만든다.

실제 데이터는 이 연결 소켓을 통해 주고받는다.

그래서 하나의 웹 서버 프로세스는 여러 클라이언트와 동시에 TCP 연결을 유지할 수 있다.

각 연결은 4-튜플로 구분된다.

## 3.2.7 웹 서버와 연결 관리

웹 서버는 많은 클라이언트의 요청을 동시에 처리해야 한다.

TCP 관점에서 보면 클라이언트마다 별도의 연결이 만들어질 수 있다.

각 연결은 다음 정보를 통해 구분된다.

- 클라이언트 IP 주소
- 클라이언트 포트 번호
- 서버 IP 주소
- 서버 포트 번호

예를 들어 두 클라이언트가 같은 웹 서버의 80번 포트에 접속한다고 하자.

두 클라이언트의 목적지 IP와 목적지 포트는 같을 수 있다.

하지만 출발지 IP 또는 출발지 포트가 다르므로 서버는 두 연결을 구분할 수 있다.

이 구조 덕분에 하나의 서버가 동시에 많은 클라이언트를 처리할 수 있다.

## 3.2.8 UDP와 TCP의 차이 정리

UDP와 TCP는 역다중화 방식에서 차이가 있다.

| 구분 | UDP | TCP |
| --- | --- | --- |
| 소켓 식별 | 목적지 IP, 목적지 포트 중심 | 출발지 IP, 출발지 포트, 목적지 IP, 목적지 포트 |
| 연결 여부 | 비연결형 | 연결형 |
| 같은 목적지 포트 | 같은 소켓으로 전달 가능 | 연결별로 다른 소켓 가능 |
| 예시 | DNS 요청 처리 | 웹 서버의 다중 클라이언트 처리 |

UDP는 단순하게 목적지 포트 중심으로 데이터를 전달한다.

TCP는 연결을 구분해야 하므로 4-튜플을 사용한다.

## 정리

다중화는 송신 측에서 여러 애플리케이션의 데이터를 모아 트랜스포트 세그먼트로 만드는 과정이다.

역다중화는 수신 측에서 도착한 세그먼트를 올바른 소켓으로 전달하는 과정이다.

포트 번호는 이 과정에서 핵심적인 역할을 한다.

UDP는 목적지 포트 중심으로 소켓을 찾고, TCP는 출발지와 목적지의 IP 주소 및 포트 번호를 모두 사용해 연결을 구분한다.

이 기능 덕분에 하나의 호스트에서 여러 네트워크 애플리케이션이 동시에 자연스럽게 동작할 수 있다.
