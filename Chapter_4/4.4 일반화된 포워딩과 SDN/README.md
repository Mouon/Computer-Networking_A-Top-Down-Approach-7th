# 4.4 일반화된 포워딩과 SDN

지금까지 살펴본 전통적인 라우터 포워딩은 주로 목적지 IP 주소를 기준으로 동작했다.

즉, 라우터는 패킷의 목적지 주소를 보고 포워딩 테이블에서 가장 잘 맞는 항목을 찾은 뒤, 해당 출력 포트로 패킷을 보냈다.

하지만 네트워크 장비가 반드시 목적지 주소만 보고 동작해야 하는 것은 아니다.

패킷 헤더의 여러 필드를 함께 보고 더 다양한 동작을 할 수도 있다.

이런 방식을 **일반화된 포워딩(generalized forwarding)** 이라고 한다.

SDN에서는 일반화된 포워딩이 중요한 개념으로 사용된다.

## 4.4.1 목적지 기반 포워딩의 한계

목적지 기반 포워딩은 단순하고 효율적이다.

하지만 표현할 수 있는 정책이 제한적이다.

예를 들어 다음과 같은 요구사항을 생각해보자.

- 특정 출발지에서 온 트래픽은 방화벽을 거치게 하고 싶다.
- 특정 애플리케이션 트래픽만 다른 경로로 보내고 싶다.
- TCP 트래픽과 UDP 트래픽을 다르게 처리하고 싶다.
- 의심스러운 패킷은 버리고 싶다.
- 특정 플로우에 대해 부하 분산을 하고 싶다.

이런 정책은 목적지 주소 하나만으로는 표현하기 어렵다.

그래서 더 일반적인 형태의 포워딩 규칙이 필요하다.

## 4.4.2 매치 플러스 액션

일반화된 포워딩의 기본 구조는 **매치 플러스 액션(match plus action)** 이다.

라우터나 스위치는 패킷 헤더의 여러 필드를 검사한다.

그리고 특정 조건과 매칭되면 정해진 동작을 수행한다.

구조는 다음과 같다.

```text
match: 패킷 헤더 필드 조건
action: 조건에 맞는 패킷에 수행할 동작
```

예를 들어 다음과 같은 규칙을 생각할 수 있다.

```text
match: 목적지 IP 주소가 10.2.0.3
action: 3번 포트로 전달
```

또는 다음과 같은 규칙도 가능하다.

```text
match: TCP 목적지 포트가 80
action: 방화벽으로 전달
```

이 방식은 단순 포워딩보다 훨씬 유연하다.

## 4.4.3 매치

**매치(match)** 는 패킷의 어떤 필드를 검사할지 정의한다.

OpenFlow 같은 SDN 기술에서는 여러 계층의 헤더 필드를 매치 조건으로 사용할 수 있다.

예를 들면 다음과 같다.

- 입력 포트
- 이더넷 출발지 주소
- 이더넷 목적지 주소
- 이더넷 타입
- IP 출발지 주소
- IP 목적지 주소
- IP 프로토콜 번호
- TCP/UDP 출발지 포트
- TCP/UDP 목적지 포트

전통적인 IP 포워딩이 목적지 IP 주소만 주로 보던 것과 비교하면 훨씬 넓은 조건을 사용할 수 있다.

예를 들어 다음 조건은 특정 웹 트래픽을 의미할 수 있다.

```text
IP 목적지 주소 = 10.1.2.3
TCP 목적지 포트 = 80
```

또 다음 조건은 특정 호스트에서 온 UDP 트래픽을 의미할 수 있다.

```text
IP 출발지 주소 = 10.0.0.5
IP 프로토콜 = UDP
```

## 4.4.4 액션

**액션(action)** 은 매치된 패킷에 대해 수행할 동작이다.

대표적인 액션은 다음과 같다.

### 포워딩

패킷을 특정 출력 포트로 보낸다.

전통적인 라우터 포워딩과 가장 비슷한 동작이다.

### 드롭

패킷을 버린다.

방화벽이나 접근 제어 정책을 구현할 때 사용할 수 있다.

### 필드 수정

패킷 헤더의 특정 필드를 수정한다.

예를 들어 NAT처럼 주소나 포트 번호를 바꾸는 동작이 가능하다.

### 컨트롤러로 전달

패킷을 SDN 컨트롤러로 보낼 수 있다.

스위치가 처리 방법을 모르는 새로운 플로우를 만났을 때, 컨트롤러에게 판단을 요청할 수 있다.

## 4.4.5 OpenFlow

OpenFlow는 SDN에서 널리 알려진 프로토콜 중 하나이다.

OpenFlow 스위치는 **플로우 테이블(flow table)** 을 가지고 있다.

플로우 테이블의 각 항목은 대략 다음과 같은 구성요소를 가진다.

1. 매치 필드
2. 카운터
3. 액션

### 매치 필드

패킷이 어떤 조건에 해당하는지 검사한다.

### 카운터

해당 규칙과 매칭된 패킷 수나 바이트 수를 기록한다.

네트워크 모니터링이나 통계 수집에 활용할 수 있다.

### 액션

매칭된 패킷에 대해 수행할 동작을 정의한다.

포워딩, 드롭, 수정, 컨트롤러 전달 등이 가능하다.

## 4.4.6 일반화된 포워딩으로 가능한 기능

일반화된 포워딩은 단순한 라우터 이상의 기능을 표현할 수 있다.

### 라우터

목적지 IP 주소를 기준으로 출력 포트를 결정하면 전통적인 라우터처럼 동작한다.

```text
match: 목적지 IP 프리픽스
action: 특정 출력 포트로 전달
```

### 스위치

이더넷 목적지 주소를 기준으로 출력 포트를 결정하면 링크 계층 스위치처럼 동작한다.

```text
match: 이더넷 목적지 주소
action: 특정 출력 포트로 전달
```

### 방화벽

특정 조건의 패킷을 버리면 방화벽 기능을 구현할 수 있다.

```text
match: TCP 목적지 포트 = 22
action: drop
```

### NAT

패킷의 IP 주소나 포트 번호를 수정하면 NAT와 비슷한 기능을 구현할 수 있다.

```text
match: 내부 출발지 주소
action: 출발지 주소와 포트 수정
```

### 부하 분산

특정 서비스로 가는 트래픽을 여러 서버로 나누어 보낼 수도 있다.

```text
match: 목적지 IP = 서비스 VIP
action: 여러 서버 중 하나로 전달
```

즉, 일반화된 포워딩은 라우터, 스위치, 방화벽, NAT, 로드밸런서 같은 여러 기능을 하나의 규칙 모델로 표현할 수 있게 한다.

## 4.4.7 SDN에서의 의미

SDN의 핵심 아이디어는 데이터 평면과 제어 평면을 분리하는 것이다.

스위치나 라우터는 데이터 평면에서 빠르게 패킷을 처리한다.

컨트롤러는 제어 평면에서 전체 네트워크 상태와 정책을 보고 플로우 테이블을 설정한다.

일반화된 포워딩은 이 구조와 잘 맞는다.

컨트롤러는 네트워크 정책을 매치 플러스 액션 규칙으로 바꾸어 스위치에 설치할 수 있다.

스위치는 설치된 규칙에 따라 패킷을 빠르게 처리한다.

이 방식의 장점은 다음과 같다.

1. 네트워크 정책을 중앙에서 관리하기 쉽다.
2. 새로운 기능을 소프트웨어적으로 추가하기 쉽다.
3. 장비별 설정을 직접 만지는 부담을 줄일 수 있다.
4. 트래픽 엔지니어링과 보안 정책 적용이 유연해진다.

## 4.4.8 OpenFlow 예시

간단한 예시를 생각해보자.

호스트 h3에서 h4로 가는 트래픽을 스위치 s2의 4번 포트로 보내고 싶다면 다음과 같은 규칙을 만들 수 있다.

```text
match: 입력 포트 = 2, 목적지 IP = h4
action: 4번 포트로 전달
```

또 h3에서 h5로 가는 트래픽은 다른 포트로 보낼 수 있다.

```text
match: 입력 포트 = 2, 목적지 IP = h5
action: 3번 포트로 전달
```

이처럼 같은 스위치라도 패킷의 헤더 필드에 따라 서로 다른 동작을 하게 만들 수 있다.

더 나아가 특정 포트의 트래픽을 버리거나, 컨트롤러로 보내거나, 헤더를 수정하는 것도 가능하다.

## 정리

전통적인 목적지 기반 포워딩은 목적지 IP 주소를 보고 출력 포트를 결정한다.

일반화된 포워딩은 패킷 헤더의 여러 필드를 매치 조건으로 사용하고, 매칭된 패킷에 다양한 액션을 적용한다.

OpenFlow의 플로우 테이블은 매치 필드, 카운터, 액션으로 구성된다.

이 구조를 통해 라우터, 스위치, 방화벽, NAT, 부하 분산 같은 기능을 하나의 규칙 모델로 표현할 수 있다.

SDN에서는 컨트롤러가 이런 규칙을 계산하고 스위치에 설치함으로써 네트워크를 더 유연하게 제어할 수 있다.
