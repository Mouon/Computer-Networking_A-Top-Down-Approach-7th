# 4.3 인터넷 프로토콜 IP: IPv4, 주소체계, NAT, IPv6

이번 절에서는 인터넷 네트워크 계층의 핵심인 **IP(Internet Protocol)** 를 살펴본다.

IP는 인터넷에서 호스트와 라우터가 데이터그램을 주고받기 위해 사용하는 기본 프로토콜이다.

IP는 최선형 서비스를 제공한다.

즉, 데이터그램을 목적지까지 보내기 위해 노력하지만, 도착 보장이나 순서 보장은 하지 않는다.

이번 절에서는 IPv4 데이터그램 포맷, 단편화, 주소체계, NAT, IPv6를 차례로 살펴본다.

## 4.3.1 IPv4 데이터그램 포맷

IPv4 데이터그램은 헤더와 데이터로 구성된다.

헤더에는 데이터그램을 목적지까지 전달하기 위한 여러 정보가 들어 있다.

대표적인 필드는 다음과 같다.

### 버전 번호

IP 프로토콜의 버전을 나타낸다.

IPv4라면 값은 4이다.

### 헤더 길이

IPv4 헤더는 옵션 필드를 포함할 수 있으므로 길이가 가변적이다.

헤더 길이 필드는 데이터가 어디서 시작되는지 알려준다.

일반적인 IPv4 헤더는 옵션이 없을 때 20바이트이다.

### 서비스 타입

서비스 타입 필드는 서로 다른 종류의 IP 데이터그램을 구분하기 위해 사용될 수 있다.

예를 들어 실시간 트래픽과 일반 트래픽을 다르게 처리하고 싶을 때 활용할 수 있다.

### 데이터그램 길이

헤더와 데이터를 모두 포함한 IPv4 데이터그램 전체 길이를 나타낸다.

IPv4 데이터그램의 최대 길이는 65,535바이트이다.

### 식별자, 플래그, 단편화 오프셋

이 필드들은 IP 단편화와 재조립에 사용된다.

큰 데이터그램이 여러 조각으로 나뉘어 전송될 때, 목적지 호스트는 이 필드들을 이용해 원래 데이터그램으로 다시 조립한다.

### TTL

TTL(Time To Live)은 데이터그램이 네트워크 안에서 무한히 떠도는 것을 막기 위한 필드이다.

라우터를 하나 지날 때마다 TTL 값은 1씩 감소한다.

TTL이 0이 되면 라우터는 데이터그램을 버린다.

### 상위 계층 프로토콜

데이터그램의 데이터 부분이 어떤 상위 계층 프로토콜에 전달되어야 하는지 알려준다.

예를 들어 TCP라면 6, UDP라면 17이 사용된다.

### 헤더 체크섬

IPv4 헤더에 오류가 있는지 검사한다.

라우터를 지날 때마다 TTL이 바뀌므로 헤더 체크섬도 다시 계산되어야 한다.

### 출발지 IP 주소와 목적지 IP 주소

데이터그램의 송신 호스트와 수신 호스트를 나타낸다.

라우터는 주로 목적지 IP 주소를 보고 포워딩을 수행한다.

## 4.3.2 IPv4 데이터그램 단편화

네트워크 링크마다 한 번에 운반할 수 있는 최대 프레임 크기가 다를 수 있다.

이 값을 **MTU(Maximum Transmission Unit)** 라고 한다.

예를 들어 어떤 링크의 MTU가 1500바이트라면, 이 링크는 1500바이트보다 큰 IP 데이터그램을 하나의 프레임에 담아 보낼 수 없다.

그렇다면 더 큰 데이터그램은 어떻게 해야 할까?

IPv4에서는 큰 데이터그램을 여러 개의 작은 조각으로 나눌 수 있다.

이를 **단편화(fragmentation)** 라고 한다.

각 조각은 독립적인 IP 데이터그램처럼 전송된다.

목적지 호스트는 여러 조각을 다시 모아 원래 데이터그램으로 재조립한다.

### 단편화에 사용되는 필드

IPv4 단편화에는 다음 필드가 사용된다.

- 식별자
- 플래그
- 단편화 오프셋

식별자는 같은 원래 데이터그램에서 나온 조각들을 구분하는 데 사용된다.

플래그는 뒤에 더 많은 조각이 있는지 알려준다.

단편화 오프셋은 이 조각이 원래 데이터그램의 어느 위치에 해당하는지 나타낸다.

### 단편화의 단점

단편화는 편리해 보이지만 여러 단점이 있다.

1. 라우터와 호스트의 처리 부담이 커진다.
2. 조각 중 하나만 손실되어도 전체 데이터그램을 재조립할 수 없다.
3. 보안 장비나 방화벽 처리도 복잡해질 수 있다.

그래서 현대 네트워크에서는 가능하면 단편화를 피하는 방향으로 동작한다.

TCP는 보통 MSS를 조정해 IP 단편화가 발생하지 않도록 한다.

## 4.3.3 IPv4 주소체계

IPv4 주소는 32비트이다.

보통 사람이 읽기 쉽게 8비트씩 4개로 나누고, 십진수로 표현한다.

예를 들면 다음과 같다.

```text
192.168.0.1
```

각 숫자는 0부터 255 사이의 값을 가진다.

IP 주소는 호스트 자체가 아니라 **인터페이스(interface)** 에 할당된다.

라우터는 여러 링크에 연결되어 있으므로 여러 인터페이스를 가지고, 각 인터페이스마다 IP 주소를 가질 수 있다.

반면 일반 호스트는 보통 하나의 네트워크 인터페이스를 가지므로 하나의 IP 주소를 갖는 경우가 많다.

## 4.3.4 서브넷

IP 주소는 네트워크 부분과 호스트 부분으로 나눠 볼 수 있다.

같은 네트워크에 속한 인터페이스들은 보통 같은 주소 프리픽스를 공유한다.

이런 네트워크를 **서브넷(subnet)** 이라고 한다.

예를 들어 다음 주소 블록이 있다고 하자.

```text
223.1.1.0/24
```

`/24`는 앞의 24비트가 네트워크 프리픽스라는 뜻이다.

즉, `223.1.1.x` 형태의 주소들이 같은 서브넷에 속할 수 있다.

서브넷 안의 호스트들은 라우터를 거치지 않고도 직접 링크 계층을 통해 서로 통신할 수 있다.

## 4.3.5 CIDR

과거에는 IP 주소를 클래스 A, B, C로 나누어 할당했다.

하지만 이 방식은 주소 공간을 비효율적으로 사용했다.

이를 개선한 방식이 **CIDR(Classless Inter-Domain Routing)** 이다.

CIDR에서는 주소 블록을 다음과 같이 표현한다.

```text
a.b.c.d/x
```

여기서 `x`는 주소의 앞 몇 비트가 네트워크 프리픽스인지 나타낸다.

예를 들어 `200.23.16.0/20`은 앞 20비트가 공통 프리픽스인 주소 블록을 의미한다.

CIDR의 장점은 주소 블록을 더 유연하게 나눌 수 있다는 것이다.

또한 여러 주소 블록을 하나의 프리픽스로 묶어 라우팅 테이블 크기를 줄일 수 있다.

이를 경로 집약(route aggregation)이라고 한다.

## 4.3.6 DHCP

호스트가 네트워크에 접속하려면 IP 주소가 필요하다.

IP 주소를 사람이 직접 설정할 수도 있지만, 실제로는 자동으로 할당받는 경우가 많다.

이때 사용되는 프로토콜이 **DHCP(Dynamic Host Configuration Protocol)** 이다.

DHCP는 호스트에게 다음 정보를 제공할 수 있다.

- IP 주소
- 서브넷 마스크
- 기본 게이트웨이
- 로컬 DNS 서버 주소

DHCP 동작은 보통 다음 네 단계로 설명된다.

1. DHCP Discover
2. DHCP Offer
3. DHCP Request
4. DHCP ACK

### DHCP Discover

새로 접속한 호스트는 사용할 IP 주소가 없으므로 브로드캐스트로 DHCP 서버를 찾는다.

### DHCP Offer

DHCP 서버는 사용할 수 있는 IP 주소와 설정 정보를 제안한다.

### DHCP Request

호스트는 제안받은 주소를 사용하겠다고 요청한다.

### DHCP ACK

DHCP 서버는 최종적으로 주소 할당을 승인한다.

DHCP 덕분에 사용자는 네트워크에 접속할 때 IP 주소를 직접 입력하지 않아도 된다.

## 4.3.7 NAT

IPv4 주소는 32비트이므로 주소 공간에 한계가 있다.

인터넷에 연결되는 장치가 폭발적으로 늘어나면서 공인 IPv4 주소가 부족해졌다.

이 문제를 완화하기 위해 널리 사용된 기술이 **NAT(Network Address Translation)** 이다.

NAT는 사설 네트워크 내부에서는 사설 IP 주소를 사용하고, 외부 인터넷과 통신할 때는 하나의 공인 IP 주소를 공유하게 해준다.

예를 들어 집 안의 장치들이 다음 주소를 사용한다고 하자.

```text
10.0.0.1
10.0.0.2
10.0.0.3
```

이 주소들은 사설 주소이므로 인터넷 전체에서 직접 라우팅되지 않는다.

집의 NAT 라우터는 외부로 나가는 패킷의 출발지 주소와 포트 번호를 자신의 공인 IP 주소와 새로운 포트 번호로 바꾼다.

그리고 NAT 변환 테이블에 이 매핑을 저장한다.

응답 패킷이 돌아오면 NAT 라우터는 변환 테이블을 보고 다시 내부 호스트의 주소와 포트로 바꿔 전달한다.

### NAT의 장점

- 여러 내부 장치가 하나의 공인 IP 주소를 공유할 수 있다.
- 내부 네트워크 주소 구조를 외부에 숨길 수 있다.
- 내부 장치 주소를 바꿔도 외부 인터넷 구성에는 영향이 적다.

### NAT의 단점

NAT는 편리하지만 논란도 있다.

원래 인터넷의 종단 간 원칙에서는 호스트들이 직접 서로 통신할 수 있어야 한다.

하지만 NAT 뒤의 호스트는 외부에서 직접 접근하기 어렵다.

또한 NAT는 IP 주소뿐 아니라 포트 번호까지 수정하므로 계층 구조를 흐린다는 비판도 있다.

그럼에도 IPv4 주소 부족 문제 때문에 NAT는 매우 널리 사용되고 있다.

## 4.3.8 IPv6

IPv4 주소 부족 문제를 근본적으로 해결하기 위해 등장한 것이 **IPv6** 이다.

IPv6 주소는 128비트이다.

IPv4의 32비트 주소보다 훨씬 큰 주소 공간을 제공한다.

IPv6는 주소 공간 확대 외에도 헤더 구조를 단순화하고, 불필요한 기능을 제거했다.

### IPv6 데이터그램 포맷

IPv6의 기본 헤더는 40바이트로 고정되어 있다.

IPv4와 달리 헤더 길이가 고정되어 있으므로 라우터 처리가 더 단순해질 수 있다.

IPv6 헤더의 주요 필드는 다음과 같다.

- 버전
- 트래픽 클래스
- 플로우 라벨
- 페이로드 길이
- 다음 헤더
- 홉 제한
- 출발지 주소
- 목적지 주소

IPv4의 TTL에 해당하는 기능은 IPv6에서 홉 제한(hop limit) 필드가 담당한다.

### IPv6에서 달라진 점

IPv6는 IPv4와 비교해 다음 특징을 가진다.

1. 주소 길이가 128비트이다.
2. 헤더 길이가 고정되어 있다.
3. 라우터에서 단편화를 수행하지 않는다.
4. 헤더 체크섬이 없다.
5. 옵션은 확장 헤더로 처리한다.

라우터가 단편화를 하지 않기 때문에 송신 호스트가 경로 MTU를 고려해야 한다.

헤더 체크섬을 제거한 것도 라우터 처리 속도를 높이기 위한 선택이다.

링크 계층과 트랜스포트 계층에서도 오류 검출을 하기 때문이다.

## 4.3.9 IPv4에서 IPv6로의 전환

IPv6가 더 큰 주소 공간을 제공하지만, 인터넷 전체를 한 번에 IPv6로 바꿀 수는 없다.

이미 수많은 IPv4 장비와 네트워크가 존재하기 때문이다.

그래서 IPv4와 IPv6는 오랫동안 함께 사용된다.

전환 방식 중 하나는 **터널링(tunneling)** 이다.

IPv6 데이터그램을 IPv4 데이터그램의 데이터 부분에 넣어 IPv4 네트워크를 통과시키는 방식이다.

IPv6를 지원하는 라우터 사이에 IPv4 네트워크가 끼어 있어도, 터널링을 사용하면 IPv6 데이터그램을 전달할 수 있다.

## 정리

IP는 인터넷 네트워크 계층의 핵심 프로토콜이다.

IPv4 데이터그램은 목적지 주소, TTL, 상위 계층 프로토콜, 체크섬 같은 필드를 가진다.

IPv4에서는 MTU보다 큰 데이터그램을 단편화할 수 있지만, 단편화는 가능하면 피하는 것이 좋다.

IPv4 주소는 인터페이스에 할당되고, CIDR을 통해 유연한 주소 블록과 경로 집약이 가능하다.

DHCP는 호스트에게 IP 주소와 네트워크 설정을 자동으로 제공한다.

NAT는 여러 사설 주소 장치가 하나의 공인 IPv4 주소를 공유하게 해준다.

IPv6는 128비트 주소와 단순화된 헤더를 통해 IPv4의 한계를 해결하려는 프로토콜이다.
