# 2.4 DNS - 인터넷의 디렉터리 서비스

인터넷에서 사람은 보통 `www.google.com` 같은 **호스트 이름(hostname)**을 사용한다.

하지만 네트워크 장비는 실제로 데이터를 보낼 때 IP 주소를 사용한다.

즉, 사람에게 편한 이름과 라우터가 이해하는 주소가 다르다.

이 둘을 연결해주는 시스템이 바로 **DNS(Domain Name System)**이다.

DNS는 인터넷의 디렉터리 서비스라고 볼 수 있다.

## 2.4.1 DNS가 제공하는 서비스

DNS의 가장 대표적인 기능은 **호스트 이름을 IP 주소로 변환**하는 것이다.

예를 들어 브라우저에 다음 주소를 입력했다고 하자.

```text
www.example.com
```

브라우저는 이 이름만으로는 서버에 직접 접속할 수 없다. 먼저 DNS를 통해 이 호스트 이름에 대응하는 IP 주소를 알아낸다.

그 후 해당 IP 주소로 TCP 연결을 만들고 HTTP 요청을 보낸다.

DNS는 단순히 이름을 IP 주소로 바꾸는 것 외에도 여러 서비스를 제공한다.

### 호스트 별칭

어떤 호스트는 복잡한 정식 이름을 가질 수 있다.

예를 들어 실제 서버 이름은 길고 복잡하지만, 사용자는 기억하기 쉬운 별칭을 사용할 수 있다.

DNS는 별칭 이름을 정식 이름으로 매핑할 수 있다.

### 메일 서버 별칭

메일 주소는 보통 다음과 같이 단순한 형태를 가진다.

```text
user@example.com
```

하지만 실제 메일 서버의 호스트 이름은 이보다 더 복잡할 수 있다.

DNS는 메일 도메인에 대해 어떤 메일 서버가 메일을 받아야 하는지도 알려준다.

이때 사용되는 레코드가 **MX 레코드**이다.

### 부하 분산

하나의 인기 있는 서비스는 여러 서버에 복제되어 있을 수 있다.

예를 들어 `www.example.com`이라는 이름이 여러 IP 주소에 대응될 수 있다.

DNS는 클라이언트에게 여러 IP 주소 중 하나를 알려주면서 트래픽을 여러 서버로 나눌 수 있다.

이런 방식으로 DNS는 간단한 형태의 부하 분산에도 사용된다.

## 2.4.2 DNS는 왜 분산 구조일까?

처음 생각해볼 수 있는 방법은 전 세계 모든 호스트 이름과 IP 주소를 하나의 거대한 DNS 서버에 저장하는 것이다.

하지만 이런 중앙 집중 방식은 현실적으로 적합하지 않다.

이유는 다음과 같다.

1. **단일 장애 지점**이 된다.
2. 전 세계 요청이 한 서버로 몰려 **트래픽이 집중**된다.
3. 먼 지역의 사용자는 응답 시간이 길어진다.
4. 모든 호스트 정보를 한곳에서 관리하기 어렵다.

그래서 DNS는 중앙 집중식 데이터베이스가 아니라 **분산 계층 데이터베이스**로 설계되었다.

## 2.4.3 DNS 서버 계층 구조

DNS 서버는 크게 다음 계층으로 나눌 수 있다.

1. **루트 DNS 서버(root DNS server)**
2. **최상위 도메인 DNS 서버(TLD DNS server)**
3. **책임 DNS 서버(authoritative DNS server)**

### 루트 DNS 서버

루트 DNS 서버는 DNS 계층 구조의 최상위에 있다.

루트 서버는 모든 호스트 이름의 IP 주소를 직접 알고 있지는 않다.

대신 `.com`, `.org`, `.edu`, `.kr` 같은 최상위 도메인을 담당하는 TLD 서버의 위치를 알려준다.

### TLD DNS 서버

TLD DNS 서버는 최상위 도메인을 담당한다.

예를 들어 `.com` TLD 서버는 `google.com`, `example.com` 같은 도메인에 대한 다음 단계 정보를 제공한다.

마찬가지로 `.kr` TLD 서버는 `co.kr`, `ac.kr` 같은 도메인과 관련된 정보를 제공한다.

### 책임 DNS 서버

책임 DNS 서버는 특정 조직이나 도메인에 대한 실제 DNS 레코드를 관리한다.

예를 들어 `example.com` 도메인의 책임 DNS 서버는 다음과 같은 정보를 가지고 있을 수 있다.

- `www.example.com`의 IP 주소
- `mail.example.com`의 IP 주소
- `example.com`의 메일 서버 정보

사용자가 최종적으로 원하는 IP 주소는 보통 이 책임 DNS 서버에서 얻는다.

## 2.4.4 로컬 DNS 서버

DNS 계층 구조에는 루트, TLD, 책임 DNS 서버가 있지만, 실제 사용자는 보통 이 서버들에게 직접 질의하지 않는다.

사용자의 호스트는 먼저 **로컬 DNS 서버(local DNS server)**에 질의한다.

로컬 DNS 서버는 일반적으로 다음과 같은 곳에서 제공한다.

- ISP
- 학교
- 회사
- 공용 DNS 서비스

브라우저가 어떤 호스트 이름의 IP 주소를 알고 싶을 때, 먼저 로컬 DNS 서버에 요청한다.

로컬 DNS 서버가 답을 캐시에 가지고 있다면 바로 응답한다.

캐시에 없다면 로컬 DNS 서버가 루트 DNS 서버부터 시작해 필요한 DNS 서버들을 차례로 질의한다.

## 2.4.5 DNS 질의 과정

예를 들어 사용자가 `www.example.com`의 IP 주소를 알고 싶다고 하자.

전체 과정은 다음과 같이 진행될 수 있다.

1. 사용자 호스트가 로컬 DNS 서버에 `www.example.com`의 IP 주소를 요청한다.
2. 로컬 DNS 서버가 루트 DNS 서버에 질의한다.
3. 루트 DNS 서버는 `.com` TLD DNS 서버를 알려준다.
4. 로컬 DNS 서버가 `.com` TLD DNS 서버에 질의한다.
5. TLD DNS 서버는 `example.com`의 책임 DNS 서버를 알려준다.
6. 로컬 DNS 서버가 책임 DNS 서버에 질의한다.
7. 책임 DNS 서버가 `www.example.com`의 IP 주소를 알려준다.
8. 로컬 DNS 서버가 사용자 호스트에게 IP 주소를 반환한다.

이후 사용자 호스트는 반환받은 IP 주소를 사용해 서버에 접속한다.

### 반복 질의와 재귀 질의

DNS 질의 방식은 크게 두 가지로 설명할 수 있다.

- **반복 질의(iterative query)**
- **재귀 질의(recursive query)**

반복 질의에서는 DNS 서버가 최종 답을 모르면, 다음에 물어볼 서버를 알려준다.

재귀 질의에서는 질의를 받은 DNS 서버가 다른 DNS 서버에 대신 질의해서 최종 답을 구한 뒤 응답한다.

실제 DNS 시스템에서는 이 두 방식이 상황에 따라 함께 사용된다.

## 2.4.6 DNS 캐싱

DNS는 성능을 높이고 트래픽을 줄이기 위해 **캐싱(caching)**을 적극적으로 사용한다.

로컬 DNS 서버가 어떤 호스트 이름의 IP 주소를 한 번 알아냈다면, 그 결과를 일정 시간 동안 저장해둔다.

이후 같은 질의가 들어오면 다시 루트 서버부터 질의하지 않고 캐시된 값을 바로 반환할 수 있다.

캐싱의 장점은 다음과 같다.

1. DNS 응답 시간이 줄어든다.
2. 루트 DNS 서버와 TLD DNS 서버의 부하가 줄어든다.
3. 반복되는 요청에 대한 네트워크 트래픽이 감소한다.

다만 캐시된 정보가 영원히 유지되면 안 된다.

도메인의 IP 주소가 바뀔 수도 있기 때문이다.

그래서 DNS 레코드는 **TTL(Time To Live)** 값을 가진다. TTL은 해당 레코드를 캐시에 얼마나 오래 보관할 수 있는지를 나타낸다.

## 2.4.7 DNS 레코드

DNS 데이터베이스는 **자원 레코드(resource record)**로 구성된다.

자원 레코드는 다음 4개의 필드를 가진다.

```text
(Name, Value, Type, TTL)
```

여기서 `Type`에 따라 `Name`과 `Value`의 의미가 달라진다.

### A 레코드

`Type=A`인 레코드는 호스트 이름을 IP 주소로 매핑한다.

```text
(www.example.com, 93.184.216.34, A)
```

이 레코드는 `www.example.com`의 IP 주소가 `93.184.216.34`라는 뜻이다.

### NS 레코드

`Type=NS`인 레코드는 어떤 도메인의 책임 DNS 서버를 알려준다.

```text
(example.com, dns.example.com, NS)
```

이 레코드는 `example.com` 도메인의 DNS 정보를 `dns.example.com`이 관리한다는 뜻이다.

### CNAME 레코드

`Type=CNAME`인 레코드는 별칭 이름을 정식 이름으로 매핑한다.

```text
(www.example.com, server-east.example.com, CNAME)
```

이 레코드는 `www.example.com`이 `server-east.example.com`의 별칭이라는 뜻이다.

### MX 레코드

`Type=MX`인 레코드는 메일 서버를 알려준다.

```text
(example.com, mail.example.com, MX)
```

이 레코드는 `example.com`으로 오는 메일을 `mail.example.com`이 처리한다는 뜻이다.

## 2.4.8 DNS 메시지

DNS 질의 메시지와 응답 메시지는 같은 기본 포맷을 사용한다.

DNS 메시지는 크게 다음과 같은 필드로 구성된다.

- 식별자
- 플래그
- 질문 영역
- 응답 영역
- 책임 영역
- 추가 영역

식별자는 질의와 응답을 매칭하기 위해 사용된다.

질문 영역에는 질의하려는 이름과 타입이 들어간다.

응답 영역에는 질의에 대한 자원 레코드가 들어간다.

## 2.4.9 DNS에 레코드 등록하기

새로운 도메인을 인터넷에서 사용하려면 도메인 등록기관(registrar)을 통해 도메인을 등록해야 한다.

도메인을 등록할 때는 보통 다음 정보를 함께 제공한다.

- 도메인 이름
- 책임 DNS 서버의 이름
- 책임 DNS 서버의 IP 주소

등록기관은 이 정보를 TLD DNS 서버에 반영한다.

이후 사용자가 해당 도메인에 대해 DNS 질의를 하면, TLD DNS 서버가 책임 DNS 서버를 알려줄 수 있다.

그리고 책임 DNS 서버에는 실제 웹 서버나 메일 서버에 대한 A, MX 등의 레코드를 등록한다.

## 정리

DNS는 사람이 이해하기 쉬운 호스트 이름을 네트워크가 사용하는 IP 주소로 변환하는 시스템이다.

단일 서버가 아니라 루트 DNS 서버, TLD DNS 서버, 책임 DNS 서버, 로컬 DNS 서버가 함께 동작하는 분산 계층 구조로 이루어져 있다.

DNS는 캐싱을 통해 응답 시간을 줄이고, A, NS, CNAME, MX 같은 자원 레코드를 통해 다양한 이름 매핑 기능을 제공한다.

웹 접속, 메일 전달, CDN 서버 선택 등 많은 인터넷 서비스가 DNS 위에서 자연스럽게 동작한다.
