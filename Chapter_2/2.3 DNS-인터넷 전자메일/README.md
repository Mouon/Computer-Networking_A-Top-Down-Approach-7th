# 2.3 인터넷 전자메일

이번 절에서는 애플리케이션 계층에서 매우 오래전부터 사용되어 온 대표적인 서비스인 **전자메일(e-mail)**을 살펴본다.

전자메일은 겉으로 보기에는 사용자가 메일을 작성하고 전송 버튼을 누르면 끝나는 단순한 서비스처럼 보인다.

하지만 실제로는 여러 애플리케이션 계층 구성요소가 함께 동작한다.

전자메일 시스템은 크게 다음 3가지 요소로 구성된다.

1. **사용자 에이전트(user agent)**
2. **메일 서버(mail server)**
3. **SMTP(Simple Mail Transfer Protocol)**

## 2.3.1 인터넷 전자메일의 구성 요소

### 사용자 에이전트

사용자 에이전트는 사용자가 메일을 읽고, 작성하고, 답장하고, 전달할 수 있게 해주는 프로그램이다.

대표적인 예시는 다음과 같다.

- Gmail, Outlook 같은 웹 메일 서비스
- Apple Mail 같은 데스크톱 메일 클라이언트
- 스마트폰의 메일 앱

사용자가 메일을 작성한 뒤 전송하면, 사용자 에이전트는 이 메일을 사용자의 **메일 서버**로 보낸다.

### 메일 서버

메일 서버는 전자메일 시스템의 중심이다.

각 수신자는 자신의 메일 서버 안에 **메일박스(mailbox)**를 갖는다. 이 메일박스는 수신자에게 도착한 메시지를 저장하고 관리한다.

예를 들어 Alice가 Bob에게 메일을 보낸다고 생각해보자.

1. Alice의 사용자 에이전트가 메일을 Alice의 메일 서버로 보냄
2. Alice의 메일 서버가 Bob의 메일 서버로 메일을 전달함
3. Bob의 메일 서버는 Bob의 메일박스에 메일을 저장함
4. Bob은 사용자 에이전트를 통해 자신의 메일박스에서 메일을 읽음

여기서 중요한 점은 송신자와 수신자가 항상 동시에 온라인일 필요가 없다는 것이다.

수신자가 오프라인이어도 메일은 수신자의 메일 서버에 저장된다. 이후 수신자가 접속하면 저장된 메일을 읽을 수 있다.

### SMTP

SMTP는 인터넷 전자메일에서 메일 서버 사이의 메시지 전달을 담당하는 애플리케이션 계층 프로토콜이다.

SMTP는 **TCP**를 사용한다. 일반적으로 SMTP 서버는 **25번 포트**에서 연결을 기다린다.

SMTP는 다음과 같은 특징을 가진다.

- 송신자의 메일 서버에서 수신자의 메일 서버로 메시지를 보낸다.
- 신뢰적인 데이터 전송을 위해 TCP를 사용한다.
- 클라이언트가 서버에게 데이터를 밀어 넣는 **푸시(push) 프로토콜**이다.
- 메시지 전송 과정에서 명령과 응답이 ASCII 텍스트 형태로 오간다.

HTTP가 서버에서 객체를 가져오는 **풀(pull) 프로토콜**이라면, SMTP는 송신 메일 서버가 수신 메일 서버로 메시지를 보내는 **푸시 프로토콜**이라고 볼 수 있다.

## 2.3.2 SMTP 동작 방식

SMTP는 클라이언트와 서버가 서로 명령과 응답을 주고받으며 메시지를 전송한다.

Alice가 Bob에게 메일을 보내는 과정을 SMTP 관점에서 보면 다음과 같다.

1. Alice의 메일 서버가 Bob의 메일 서버에 TCP 연결을 연다.
2. Bob의 메일 서버가 연결을 수락한다.
3. Alice의 메일 서버는 SMTP 명령을 통해 송신자와 수신자를 알린다.
4. Alice의 메일 서버는 메일 본문을 전송한다.
5. 전송이 끝나면 TCP 연결을 닫는다.

SMTP 대화는 대략 다음과 같은 형태를 가진다.

```text
S: 220 hamburger.edu
C: HELO crepes.fr
S: 250 Hello crepes.fr
C: MAIL FROM:<alice@crepes.fr>
S: 250 alice@crepes.fr ... Sender ok
C: RCPT TO:<bob@hamburger.edu>
S: 250 bob@hamburger.edu ... Recipient ok
C: DATA
S: 354 Enter mail, end with "." on a line by itself
C: Do you like ketchup?
C: .
S: 250 Message accepted for delivery
C: QUIT
S: 221 hamburger.edu closing connection
```

위 예시에서 `C`는 클라이언트, `S`는 서버를 의미한다.

메일 본문은 `DATA` 명령 이후에 전송된다. 그리고 본문이 끝났다는 것을 알리기 위해 한 줄에 마침표(`.`)만 있는 줄을 보낸다.

SMTP는 하나의 TCP 연결 안에서 여러 개의 메일 메시지를 보낼 수도 있다. 같은 수신 메일 서버로 여러 메시지를 보내야 한다면 매번 TCP 연결을 새로 만들 필요가 없다.

## 2.3.3 HTTP와 SMTP 비교

HTTP와 SMTP는 모두 애플리케이션 계층 프로토콜이고, 둘 다 TCP를 사용한다.

하지만 두 프로토콜의 목적과 동작 방식은 다르다.

### HTTP

HTTP는 웹 브라우저가 웹 서버에서 객체를 가져오기 위해 사용한다.

즉, 사용자가 웹 페이지를 요청하면 브라우저가 서버에게 객체를 달라고 요청하고, 서버는 응답으로 객체를 보낸다.

이 때문에 HTTP는 **풀 프로토콜**에 가깝다.

### SMTP

SMTP는 송신 메일 서버가 수신 메일 서버로 메시지를 밀어 넣기 위해 사용한다.

메일을 받는 쪽이 먼저 가져가는 것이 아니라, 보내는 쪽이 수신자의 메일 서버로 전달한다.

이 때문에 SMTP는 **푸시 프로토콜**에 가깝다.

### 차이점 정리

- HTTP는 웹 객체를 가져오는 데 사용된다.
- SMTP는 전자메일 메시지를 전달하는 데 사용된다.
- HTTP는 주로 클라이언트가 서버에서 데이터를 가져온다.
- SMTP는 송신 메일 서버가 수신 메일 서버로 데이터를 보낸다.
- HTTP 메시지는 각 객체가 별도의 응답 메시지로 캡슐화된다.
- SMTP는 여러 객체가 포함된 메시지를 하나의 메일 메시지 안에 넣어 보낸다.

## 2.3.4 메일 메시지 포맷

전자메일 메시지는 크게 두 부분으로 구성된다.

1. **헤더(header)**
2. **본문(body)**

헤더에는 메일을 처리하는 데 필요한 부가 정보가 들어간다.

예를 들면 다음과 같다.

```text
From: alice@crepes.fr
To: bob@hamburger.edu
Subject: Searching for the meaning of life
```

헤더와 본문은 빈 줄로 구분된다.

본문에는 사용자가 실제로 작성한 메시지 내용이 들어간다.

```text
From: alice@crepes.fr
To: bob@hamburger.edu
Subject: Searching for the meaning of life

Do you like ketchup?
```

초기의 SMTP는 ASCII 텍스트 전송을 중심으로 설계되었다.

하지만 오늘날의 메일은 이미지, 첨부파일, HTML, 다양한 문자 인코딩을 포함한다.

이런 데이터를 메일 안에 표현하기 위해 **MIME(Multipurpose Internet Mail Extensions)**이 사용된다. MIME은 메일 메시지에 포함된 데이터의 타입과 인코딩 방식을 알려준다.

## 2.3.5 메일 접속 프로토콜

SMTP는 메일 서버 사이에서 메일을 전달하는 프로토콜이다.

그렇다면 수신자는 자신의 메일 서버에 저장된 메일을 어떻게 읽을까?

수신자가 메일을 읽기 위해 사용하는 방식은 크게 다음과 같다.

- **HTTP**
- **POP3**
- **IMAP**

### HTTP 기반 웹 메일

Gmail이나 네이버 메일처럼 웹 브라우저에서 메일을 읽는 경우, 사용자의 브라우저와 메일 서버 사이에는 HTTP가 사용된다.

하지만 메일 서버 사이의 전달에는 여전히 SMTP가 사용된다.

즉, 웹 메일을 사용하더라도 전체 메일 시스템에서 SMTP가 사라지는 것은 아니다.

### POP3

POP3(Post Office Protocol version 3)는 사용자가 메일 서버에서 자신의 로컬 장치로 메일을 내려받기 위해 사용되는 프로토콜이다.

POP3의 동작은 세 단계로 나눌 수 있다.

1. **인증 단계**: 사용자 이름과 비밀번호를 확인한다.
2. **트랜잭션 단계**: 사용자가 메시지를 가져오거나 삭제 표시를 한다.
3. **갱신 단계**: 연결이 종료될 때 삭제 표시된 메시지를 실제로 삭제한다.

POP3는 단순하다는 장점이 있지만, 여러 장치에서 같은 메일 상태를 자연스럽게 공유하기에는 불편하다.

예를 들어 노트북에서 메일을 내려받아 삭제했다면, 스마트폰에서는 그 메일을 다시 보기 어려울 수 있다.

### IMAP

IMAP(Internet Mail Access Protocol)은 메일을 서버에 남겨두고, 여러 장치에서 같은 메일함 상태를 관리하기에 적합하다.

IMAP은 다음과 같은 기능을 제공한다.

- 메일을 서버에 저장한 상태로 읽기
- 폴더 생성 및 이동
- 읽음/안 읽음 상태 관리
- 여러 장치에서 동일한 메일함 상태 동기화

오늘날 여러 기기에서 같은 메일 계정을 사용하는 환경에서는 POP3보다 IMAP 방식이 더 자연스럽다.

## 정리

인터넷 전자메일은 사용자 에이전트, 메일 서버, SMTP가 함께 동작하는 시스템이다.

사용자는 사용자 에이전트로 메일을 작성하고 읽는다. 메일 서버는 사용자의 메일박스를 저장하고, SMTP는 메일 서버 사이에서 메시지를 전달한다.

메일을 읽는 방식은 환경에 따라 HTTP, POP3, IMAP이 사용된다.

결국 전자메일은 단순히 "메일을 보낸다"라는 기능 뒤에서 여러 애플리케이션 계층 프로토콜이 협력하는 구조라고 볼 수 있다.
