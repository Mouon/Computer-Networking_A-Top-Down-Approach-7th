# 2.5 P2P 파일 분배

이번 절에서는 큰 파일을 여러 사용자에게 분배하는 방법을 살펴본다.

파일 분배 방식은 크게 두 가지로 나눌 수 있다.

1. **클라이언트-서버 방식**
2. **P2P(peer-to-peer) 방식**

클라이언트-서버 방식에서는 항상 서버가 파일을 보내고, 클라이언트는 파일을 받기만 한다.

반면 P2P 방식에서는 피어(peer)들이 파일을 받는 동시에 다른 피어에게 파일을 업로드한다.

즉, P2P에서는 사용자가 소비자이면서 동시에 공급자가 된다.

## 2.5.1 클라이언트-서버 파일 분배

하나의 서버가 크기 `F`인 파일을 `N`개의 클라이언트에게 분배한다고 생각해보자.

서버의 업로드 속도를 `u_s`라고 하고, 각 클라이언트의 다운로드 속도를 `d_i`라고 하자.

이때 클라이언트-서버 방식에서는 서버가 모든 클라이언트에게 파일을 직접 보내야 한다.

따라서 서버는 총 `N * F` 비트만큼 업로드해야 한다.

클라이언트-서버 방식의 최소 분배 시간은 다음 조건에 의해 제한된다.

```text
D_cs >= max { N * F / u_s, F / d_min }
```

여기서 `d_min`은 가장 느린 클라이언트의 다운로드 속도이다.

이 식에서 알 수 있는 점은 다음과 같다.

- 클라이언트 수 `N`이 늘어나면 서버가 업로드해야 할 총량도 선형적으로 증가한다.
- 서버 업로드 속도 `u_s`가 병목이 되기 쉽다.
- 가장 느린 클라이언트의 다운로드 속도도 전체 분배 시간의 하한을 만든다.

즉, 클라이언트-서버 방식은 구조가 단순하지만 많은 사용자에게 큰 파일을 배포할 때 서버 부담이 커진다.

## 2.5.2 P2P 파일 분배

P2P 방식에서는 서버가 모든 데이터를 직접 보내지 않아도 된다.

서버가 일부 피어에게 파일 조각을 보내면, 그 피어들이 다시 다른 피어에게 조각을 업로드할 수 있다.

피어 `i`의 업로드 속도를 `u_i`라고 하자.

P2P 방식에서는 서버의 업로드 속도뿐만 아니라 피어들의 업로드 속도도 전체 파일 분배에 사용된다.

P2P 방식의 최소 분배 시간은 다음 조건에 의해 제한된다.

```text
D_p2p >= max { F / u_s, F / d_min, N * F / (u_s + u_1 + u_2 + ... + u_N) }
```

각 항의 의미는 다음과 같다.

1. `F / u_s`: 서버는 최소한 파일 한 벌은 업로드해야 한다.
2. `F / d_min`: 가장 느린 피어도 파일 전체를 다운로드해야 한다.
3. `N * F / (u_s + u_1 + ... + u_N)`: 전체 시스템이 총 `N * F` 비트를 업로드해야 한다.

P2P의 핵심은 클라이언트 수가 증가할수록 다운로드 수요만 증가하는 것이 아니라, 업로드 자원도 함께 증가한다는 점이다.

그래서 P2P는 대규모 파일 분배에서 높은 확장성을 가질 수 있다.

## 2.5.3 클라이언트-서버와 P2P 비교

클라이언트-서버 방식에서는 서버가 중심이다.

사용자가 많아질수록 서버가 감당해야 할 업로드 양이 커진다.

반면 P2P 방식에서는 피어들이 서로 파일 조각을 교환한다.

사용자가 많아지면 다운로드 요청도 늘어나지만, 동시에 업로드에 참여하는 피어 수도 늘어난다.

이 차이 때문에 P2P 방식은 많은 피어가 참여할 때 특히 강점을 가진다.

### 클라이언트-서버 방식의 특징

- 구조가 단순하다.
- 중앙 서버가 전체 파일을 관리한다.
- 서버 장애가 서비스 전체에 큰 영향을 줄 수 있다.
- 사용자 수가 많아질수록 서버 비용이 증가한다.

### P2P 방식의 특징

- 피어들이 서로 데이터를 주고받는다.
- 피어가 늘어날수록 업로드 자원도 함께 늘어난다.
- 중앙 서버의 부담을 줄일 수 있다.
- 피어의 접속/이탈이 잦기 때문에 이를 고려한 프로토콜이 필요하다.

## 2.5.4 BitTorrent

P2P 파일 분배의 대표적인 예가 **BitTorrent**이다.

BitTorrent에서는 하나의 파일을 작은 조각(piece)들로 나누고, 피어들이 서로 필요한 조각을 교환한다.

BitTorrent에 참여하는 피어들의 모임을 **토렌트(torrent)**라고 부른다.

각 피어는 파일 전체를 처음부터 가지고 있지 않아도 된다.

어떤 조각을 받으면 그 조각을 다른 피어에게 다시 업로드할 수 있다.

### 토렌트 참여 과정

새로운 피어가 토렌트에 참여하면 다음과 같은 일이 일어난다.

1. 피어는 토렌트 정보를 통해 참여할 수 있는 피어 목록을 얻는다.
2. 여러 피어와 TCP 연결을 만든다.
3. 자신이 가지고 있지 않은 조각을 다른 피어에게 요청한다.
4. 자신이 가진 조각은 다른 피어에게 업로드한다.
5. 시간이 지나면서 파일 조각을 모두 모아 전체 파일을 완성한다.

피어들은 네트워크 상황에 따라 계속 들어오고 나간다.

그래서 BitTorrent는 고정된 서버-클라이언트 구조가 아니라, 동적으로 변하는 피어 집합을 기반으로 동작한다.

## 2.5.5 희귀 조각 우선

BitTorrent에서 중요한 전략 중 하나는 **희귀 조각 우선(rarest first)**이다.

피어는 자신이 없는 조각 중에서, 현재 이웃 피어들 사이에 가장 적게 존재하는 조각을 먼저 요청한다.

이 전략의 이유는 간단하다.

많이 퍼져 있는 조각은 나중에도 구하기 쉽지만, 적게 퍼져 있는 조각은 사라질 위험이 크다.

희귀 조각을 먼저 퍼뜨리면 토렌트 전체에서 조각들이 더 균등하게 분포될 수 있다.

결과적으로 특정 조각이 부족해서 다운로드가 멈추는 상황을 줄일 수 있다.

## 2.5.6 Tit-for-tat

BitTorrent는 피어들이 업로드에 참여하도록 유도해야 한다.

모든 피어가 다운로드만 하고 업로드를 하지 않으려 한다면 P2P 구조는 제대로 동작하지 않는다.

이를 막기 위해 BitTorrent는 **tit-for-tat** 전략을 사용한다.

기본 아이디어는 다음과 같다.

- 나에게 좋은 속도로 업로드해주는 피어에게 나도 업로드한다.
- 업로드를 거의 하지 않는 피어에게는 우선순위를 낮춘다.
- 가끔 새로운 피어에게도 기회를 줘서 더 좋은 교환 상대를 찾는다.

이 방식은 피어들이 서로 협력하도록 만든다.

즉, 많이 기여하는 피어가 더 좋은 다운로드 기회를 얻는다.

## 2.5.7 P2P의 장점과 한계

P2P는 대규모 파일 분배에서 강력한 확장성을 가진다.

하지만 모든 상황에서 무조건 좋은 것은 아니다.

### 장점

- 서버 업로드 부담을 줄인다.
- 피어 수가 증가해도 시스템 전체 업로드 용량이 함께 증가한다.
- 큰 파일을 많은 사용자에게 효율적으로 분배할 수 있다.

### 한계

- 피어가 자주 접속하고 이탈한다.
- 업로드에 참여하지 않으려는 피어가 있을 수 있다.
- 보안과 신뢰성 관리가 어렵다.
- 중앙 통제가 필요한 서비스에는 맞지 않을 수 있다.

## 정리

클라이언트-서버 파일 분배는 구조가 단순하지만, 많은 사용자에게 큰 파일을 배포할수록 서버의 부담이 커진다.

P2P 파일 분배는 피어들이 다운로드와 업로드를 동시에 수행한다.

이 때문에 피어 수가 늘어날수록 수요뿐 아니라 공급 자원도 함께 증가한다.

BitTorrent는 이런 P2P 구조를 실제로 구현한 대표적인 프로토콜이며, 희귀 조각 우선과 tit-for-tat 전략을 통해 효율적인 파일 분배와 참여 유도를 함께 달성한다.
