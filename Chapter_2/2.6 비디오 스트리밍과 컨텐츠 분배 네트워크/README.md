# 2.6 비디오 스트리밍과 콘텐츠 분배 네트워크

인터넷에서 비디오는 가장 많은 트래픽을 발생시키는 애플리케이션 중 하나이다.

유튜브, 넷플릭스, 라이브 방송, 온라인 강의처럼 오늘날의 많은 서비스는 비디오 스트리밍을 중심으로 동작한다.

비디오 스트리밍은 단순히 큰 파일을 다운로드하는 것과 다르다.

사용자는 전체 파일이 모두 내려받아질 때까지 기다리지 않고, 앞부분부터 바로 재생되기를 기대한다.

그래서 비디오 스트리밍에서는 다음 문제가 중요하다.

- 끊김 없이 재생할 수 있어야 한다.
- 네트워크 대역폭 변화에 적응해야 한다.
- 많은 사용자에게 동시에 제공할 수 있어야 한다.
- 서버와 네트워크 비용을 감당할 수 있어야 한다.

## 2.6.1 인터넷 비디오

비디오는 연속적인 이미지와 소리로 구성된다.

원본 비디오는 데이터 크기가 매우 크기 때문에 그대로 전송하기 어렵다.

그래서 비디오는 보통 압축되어 저장되고 전송된다.

압축률이 높을수록 필요한 대역폭은 줄어들지만, 화질이 낮아질 수 있다.

반대로 고화질 비디오는 더 많은 비트를 필요로 한다.

예를 들어 같은 영상을 다음과 같이 여러 품질로 인코딩할 수 있다.

- 낮은 화질: 낮은 비트율, 작은 화면, 적은 대역폭
- 중간 화질: 보통 비트율, 보통 화면, 보통 대역폭
- 높은 화질: 높은 비트율, 큰 화면, 많은 대역폭

비디오 스트리밍 서비스는 사용자의 네트워크 상태에 맞춰 적절한 품질을 선택해야 한다.

네트워크 속도가 충분하지 않은데 높은 화질을 고집하면 재생 버퍼가 비어서 영상이 멈추게 된다.

## 2.6.2 HTTP 스트리밍과 DASH

초기의 비디오 스트리밍은 전용 스트리밍 프로토콜을 사용하는 경우가 많았다.

하지만 오늘날에는 HTTP를 이용한 스트리밍이 널리 사용된다.

HTTP 스트리밍에서는 비디오가 웹 서버에 일반 파일처럼 저장되고, 클라이언트는 HTTP GET 요청으로 비디오 데이터를 가져온다.

클라이언트는 받은 데이터를 버퍼에 저장하면서 동시에 재생한다.

### 단순 HTTP 스트리밍의 한계

단순한 HTTP 스트리밍에서는 하나의 비디오 파일이 하나의 품질로 제공될 수 있다.

이 방식은 네트워크 상황이 일정할 때는 괜찮지만, 실제 인터넷 환경에서는 문제가 생긴다.

사용자의 대역폭은 시간에 따라 변할 수 있다.

무선 환경에서는 신호가 약해질 수 있고, 같은 네트워크를 사용하는 사람이 늘어나면 가용 대역폭이 줄어들 수 있다.

이때 고정된 품질만 제공하면 다음 문제가 발생한다.

- 대역폭이 부족하면 재생이 멈춘다.
- 대역폭이 충분해도 낮은 화질만 볼 수 있다.
- 사용자마다 다른 네트워크 환경을 반영하기 어렵다.

### DASH

이 문제를 해결하기 위해 사용되는 방식이 **DASH(Dynamic Adaptive Streaming over HTTP)**이다.

DASH에서는 하나의 비디오를 여러 품질 버전으로 인코딩한다.

그리고 각 버전을 작은 조각(chunk)으로 나누어 서버에 저장한다.

클라이언트는 현재 네트워크 상황과 버퍼 상태를 보고, 다음 조각을 어떤 품질로 받을지 결정한다.

예를 들어 네트워크가 빠를 때는 고화질 조각을 요청하고, 네트워크가 느려지면 중간 또는 저화질 조각을 요청한다.

이렇게 하면 재생이 끊기는 것을 줄이면서 가능한 좋은 화질을 제공할 수 있다.

### 매니페스트 파일

DASH 서버는 클라이언트에게 **매니페스트 파일(manifest file)**을 제공한다.

매니페스트 파일에는 다음 정보가 들어 있다.

- 비디오가 어떤 품질 버전으로 제공되는지
- 각 품질 버전의 비트율이 얼마인지
- 각 조각의 URL이 무엇인지

클라이언트는 매니페스트 파일을 읽고, 자신에게 적합한 비디오 조각을 HTTP 요청으로 가져온다.

DASH의 핵심은 품질 선택을 서버가 아니라 **클라이언트가 동적으로 수행**한다는 점이다.

## 2.6.3 콘텐츠 분배 네트워크

인기 있는 비디오 서비스를 하나의 거대한 데이터 센터에서만 제공한다고 생각해보자.

전 세계 사용자가 모두 같은 서버로 접속한다면 다음 문제가 생긴다.

1. 사용자가 서버에서 멀리 떨어져 있으면 지연이 커진다.
2. 특정 네트워크 경로에 트래픽이 집중된다.
3. 서버와 접속 링크가 병목이 된다.
4. 장애가 발생하면 많은 사용자가 영향을 받는다.

이를 해결하기 위해 사용되는 것이 **CDN(Content Distribution Network)**이다.

CDN은 콘텐츠 복사본을 여러 지역의 서버에 분산해서 저장하고, 사용자에게 가까운 서버에서 콘텐츠를 제공하는 시스템이다.

### CDN의 기본 아이디어

CDN의 기본 아이디어는 단순하다.

콘텐츠를 사용자 가까이에 배치한다.

사용자가 비디오를 요청하면 원본 서버까지 매번 가지 않고, 가까운 CDN 서버에서 비디오를 받는다.

이를 통해 다음 효과를 얻을 수 있다.

- 응답 지연 감소
- 네트워크 혼잡 완화
- 원본 서버 부하 감소
- 장애에 대한 복원력 향상

## 2.6.4 CDN 서버 배치 전략

CDN은 서버를 어디에 둘지 결정해야 한다.

대표적인 전략은 다음 두 가지이다.

1. **enter deep**
2. **bring home**

### Enter Deep

enter deep 전략은 CDN 서버를 여러 ISP 네트워크 깊숙한 곳에 배치하는 방식이다.

사용자와 매우 가까운 위치에 서버를 두기 때문에 지연을 줄일 수 있다.

하지만 많은 위치에 서버를 설치하고 관리해야 하므로 운영 복잡도가 높다.

### Bring Home

bring home 전략은 비교적 적은 수의 큰 클러스터를 인터넷 교환 지점 근처에 배치하는 방식이다.

서버 수와 운영 위치가 줄어들기 때문에 관리가 단순하다.

하지만 사용자와의 거리가 enter deep 방식보다 멀 수 있다.

CDN 사업자는 비용, 성능, 운영 복잡도 사이에서 적절한 전략을 선택한다.

## 2.6.5 CDN 동작 방식

사용자가 어떤 비디오를 보려고 할 때 CDN이 동작하는 과정을 살펴보자.

1. 사용자가 웹 사이트에 접속한다.
2. 웹 페이지에는 비디오 객체를 가리키는 URL이 포함되어 있다.
3. 이 URL은 CDN 도메인으로 연결된다.
4. 사용자의 호스트는 CDN 도메인에 대해 DNS 질의를 수행한다.
5. CDN의 DNS 시스템은 사용자에게 적절한 CDN 서버의 IP 주소를 알려준다.
6. 사용자는 선택된 CDN 서버에서 비디오 조각을 HTTP로 받아 재생한다.

여기서 DNS는 단순히 이름을 IP 주소로 바꾸는 역할만 하지 않는다.

CDN은 DNS를 이용해 사용자를 적절한 서버로 안내한다.

서버 선택에는 다음 요소가 고려될 수 있다.

- 사용자와 서버 사이의 지리적 거리
- 네트워크 지연
- 서버 부하
- 사용자의 로컬 DNS 서버 위치
- CDN 내부 정책

## 2.6.6 클러스터 선택 전략

CDN은 사용자에게 어떤 클러스터를 배정할지 결정해야 한다.

가장 단순한 방법은 지리적으로 가까운 클러스터를 선택하는 것이다.

하지만 지리적으로 가깝다고 항상 네트워크 성능이 좋은 것은 아니다.

실제 인터넷 경로는 ISP 정책, 혼잡 상태, 라우팅 구조에 따라 달라진다.

그래서 CDN은 단순 거리뿐 아니라 실시간 측정 정보를 활용하기도 한다.

예를 들어 다음 정보를 사용할 수 있다.

- 클러스터별 응답 시간
- 패킷 손실률
- 현재 서버 부하
- 사용자의 요청 위치
- 이전 요청들의 성능 데이터

좋은 CDN은 사용자가 의식하지 못하는 사이에 더 빠르고 안정적인 서버를 선택한다.

## 2.6.7 사례: Netflix, YouTube, Kankan

대규모 비디오 서비스는 각자의 방식으로 스트리밍과 콘텐츠 분배 문제를 해결한다.

### Netflix

Netflix 같은 서비스는 비디오를 여러 화질로 인코딩하고, 사용자의 네트워크 상황에 따라 적절한 품질을 선택한다.

사용자는 영상을 보는 동안 네트워크 상태가 변해도 재생이 가능한 한 끊기지 않기를 기대한다.

그래서 DASH와 같은 적응형 스트리밍 방식이 중요하다.

또한 콘텐츠를 CDN에 배치하여 사용자에게 가까운 위치에서 비디오를 제공한다.

### YouTube

YouTube는 전 세계적으로 매우 많은 사용자가 동시에 비디오를 시청한다.

이런 규모에서는 단일 서버 구조로는 서비스가 어렵다.

YouTube는 비디오를 여러 서버와 캐시에 분산하고, HTTP 기반 스트리밍을 통해 다양한 네트워크 환경의 사용자에게 콘텐츠를 제공한다.

### Kankan

Kankan 같은 서비스는 비디오 스트리밍에 P2P 요소를 결합하는 사례로 다룰 수 있다.

사용자들이 서버에서만 비디오를 받는 것이 아니라, 다른 사용자와도 일부 데이터를 교환하면 서버 부담을 줄일 수 있다.

이는 앞 절의 P2P 파일 분배 아이디어가 비디오 스트리밍에도 응용될 수 있음을 보여준다.

## 2.6.8 스트리밍에서 중요한 성능 요소

비디오 스트리밍 서비스의 품질은 단순히 평균 다운로드 속도만으로 결정되지 않는다.

중요한 요소는 다음과 같다.

### 초기 재생 지연

사용자가 재생 버튼을 누른 뒤 영상이 시작되기까지 걸리는 시간이다.

너무 길면 사용자는 답답함을 느낀다.

### 버퍼링

재생 중 데이터가 충분히 도착하지 않으면 영상이 멈춘다.

이 현상이 반복되면 사용자 경험이 크게 나빠진다.

### 화질 변화

DASH는 네트워크 상태에 따라 화질을 바꿀 수 있다.

하지만 화질이 너무 자주 변하면 사용자가 불편함을 느낄 수 있다.

### 서버 선택

사용자에게 가까우면서도 부하가 적은 서버를 선택해야 한다.

CDN의 서버 선택이 좋지 않으면 대역폭이 충분해도 지연이나 끊김이 발생할 수 있다.

## 정리

비디오 스트리밍은 큰 데이터를 단순히 내려받는 문제가 아니라, 시간에 맞춰 끊김 없이 재생하는 문제이다.

HTTP 스트리밍은 웹 인프라를 활용할 수 있다는 장점이 있고, DASH는 네트워크 상황에 맞춰 품질을 동적으로 조절한다.

CDN은 콘텐츠를 여러 지역에 분산 배치하여 사용자에게 가까운 서버에서 제공한다.

결국 현대 비디오 서비스는 DASH 같은 적응형 스트리밍과 CDN 같은 분산 인프라가 결합되어 동작한다.
